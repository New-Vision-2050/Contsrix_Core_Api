# Multi-Tenancy Implementation Guide

This document provides an overview of how multi-tenancy has been implemented in the application using the stancl/tenancy package.

## Architecture Overview

The multi-tenancy implementation follows a database-per-tenant approach, where:

1. Each company is considered a tenant
2. Each tenant has its own database
3. The central database contains company and tenant information
4. Tenant identification is done via domain/subdomain

## Components Implemented

### 1. Tenant Module

A dedicated Tenant module has been created with the following components:

- **Models**: `Tenant` model that extends the stancl/tenancy Tenant model
- **Services**: `TenantService` for tenant management and `TenantReportingService` for cross-tenant reporting
- **Controllers**: `TenantController` for tenant management and `TenantReportingController` for reporting
- **Observers**: `CompanyObserver` to automatically create tenants when companies are created
- **Routes**: API routes for tenant management and reporting

### 2. Database Configuration

- Central database connection for global data
- Tenant connection template for tenant-specific databases
- Tenant-specific migrations in `database/migrations/tenant`

### 3. Tenant Identification

- Domain/subdomain-based tenant identification
- Each tenant has a unique domain (e.g., `company-name.yourdomain.com`)

### 4. Cross-Tenant Reporting

- `TenantReportingService` provides methods for generating reports across all tenants
- API endpoints for accessing cross-tenant reports

## How It Works

### Tenant Creation

When a company is created and activated:

1. The `CompanyObserver` detects the company creation event
2. The `TenantService` creates a new tenant with the company's ID
3. A domain is created for the tenant (e.g., `company-username.yourdomain.com`)
4. A new database is created for the tenant
5. Tenant-specific migrations are run on the new database

### Tenant Identification

When a request comes in:

1. The `InitializeTenancyByDomain` middleware identifies the tenant based on the domain
2. The tenant context is initialized
3. All database operations use the tenant's database connection

### Cross-Tenant Operations

For operations that need to access data across all tenants:

1. The `TenantReportingService` provides methods for running code in each tenant's context
2. Reports can be generated by aggregating data from all tenants

## Database Structure

### Central Database

The central database contains:

- `tenants` table: Stores tenant information
- `domains` table: Stores domain information for tenants
- `companies` table: Stores company information
- Other global tables

### Tenant Databases

Each tenant database contains:

- `tenant_settings` table: Tenant-specific settings
- All other tenant-specific tables

## API Endpoints

### Tenant Management

- `GET /api/tenants`: List all tenants
- `POST /api/tenants/companies/{companyId}`: Create a new tenant for a company
- `GET /api/tenants/{id}`: Get tenant details
- `DELETE /api/tenants/{id}`: Delete a tenant

### Reporting

- `GET /api/tenants/reports/users`: Get user count by tenant
- `GET /api/tenants/reports/dashboard`: Get dashboard statistics
- `GET /api/tenants/reports/health`: Get tenant health report

## Configuration

The following configuration files have been updated:

- `config/database.php`: Added central and tenant connection configurations
- `config/tenancy.php`: Configured tenancy settings
- `modules/Tenant/Config/config.php`: Tenant module configuration

## Environment Variables

The following environment variables should be set:

```
# Central Database Configuration
DB_CONNECTION=central
DB_DRIVER=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=central_db
DB_USERNAME=root
DB_PASSWORD=

# Tenant Database Configuration
TENANT_DB_DRIVER=mysql
TENANT_DB_HOST=127.0.0.1
TENANT_DB_PORT=3306
TENANT_DB_USERNAME=root
TENANT_DB_PASSWORD=

# Tenancy Configuration
TENANCY_DOMAIN=localhost
```

## Best Practices

1. Always use the tenant context when accessing tenant-specific data
2. Use the central database for global data that should be accessible across all tenants
3. Use the reporting service for cross-tenant operations
4. Create tenant-specific migrations in `database/migrations/tenant`
5. Use the tenant middleware for tenant-specific routes
